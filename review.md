# 코드 리뷰 — Data-Anomaly-Detection

- 대상: `index.html` (단일 파일, 약 3,800줄 / 엔진 스크립트 + 메인 스크립트 2블록)
- 리뷰 방식: 전체 소스 정독 + 실제 브라우저(headless Chromium, file://) 재현 검증
- 도메인 전제: 산업단지·공장(제조업) 데이터(산단명, 기업명, 업종, 생산·고용·입주 통계, 시계열)를 내부망 PC에서 분석하는 업무 도구
- 원칙: 모든 지적은 코드 위치와 재현 근거를 확인한 것만 기재(추측 배제)

---

## 1. 구조 요약 (분석 결과)

| 블록 | 내용 |
|---|---|
| `<script id="engineScript">` | DOM 미사용 순수 엔진. Values(값 헬퍼) → ColumnAnalyzer(타입 추론) → Detectors(9종 탐지기, 구조화 findings 반환) → runAnalysis(오케스트레이션) → Worker 진입점 |
| `<script>` (메인) | Inflate/Zip → 파일 리더(xlsx=ZIP+XML, xls=OLE2+BIFF, csv=청크 파서) + gridToTable(헤더 자동 탐지) → XlsxWriter → UI(업로드/시트/설정/실행/대시보드/검토/다운로드) → 툴팁 → 비교 검증 |
| Worker 프로토콜 | in: `{columns, rows, config}` / `{type:"cancel"}` · out: `progress{pct,msg,extra{stage,algo,subPct}}` / `done{flags,reasons,findings,skipped,approxAlgos,usedAlgos,cancelled,elapsedMs,infos}` / `error` |
| 결과 구조 | 행별 `flags`(이상/""), `reasons`(문자열), `findings[r][] = {rule, category, column, evidence, evidenceText}` |

전체적으로 엔진/리더/작성기/UI의 계층 분리가 명확하고, file:// 폴백·근사 모드·취소 등 운영 경계가 코드에 명시돼 있다. 이하는 **문제점 중심**으로 기술한다.

## 2. 버그

### BUG-1. 수치 파라미터에 0 입력 시 기본값으로 무시됨 — `collectConfig()` (약 2395행) 【Minor·즉시 수정 가능】
- **원인**: `parseFloat($("p_change").value) || 100` — `0`은 falsy이므로 사용자가 0을 입력하면 100으로 대체된다. `p_zscore`, `p_contam`, 비교 탭 `p_cmp_rate`/`p_cmp_z`도 동일 패턴.
- **영향**: "증감률 0% 초과 전부 탐지" 같은 설정 불가. 사용자는 값이 무시된 것을 알 수 없음.
- **해결**: `const v = parseFloat(x); return Number.isFinite(v) ? v : 기본값;` 헬퍼로 교체. 동시에 min/max 클램프 적용(BUG-2와 함께).

### BUG-2. 수치 입력 검증이 HTML `min` 속성에만 의존 【Minor→Major 잠재·즉시 수정 가능】
- **원인**: `min="0.1"`은 스피너에만 작동. 직접 타이핑하면 음수/비정상값이 그대로 `collectConfig`로 들어간다. 예: Z-Score 임계 `-5` 입력 → `|z| > -5`가 항상 참 → **전 행 이상 판정**.
- **영향**: 사용자 실수 한 번으로 결과 전체가 무의미해짐(업무 보고서 오염).
- **해결**: `collectConfig`에서 파라미터별 [min, max] 클램프 + 클램프 발생 시 입력창 값 동기화.
```js
function num(id, def, lo, hi){ let v=parseFloat($(id).value); if(!Number.isFinite(v)) v=def; v=Math.min(hi,Math.max(lo,v)); $(id).value=v; return v; }
```

### BUG-3. 검토 일괄 적용 시 표시 필터가 '전체'로 리셋 — `rvApply` 핸들러 (약 2721행) 【Minor·즉시 수정 가능】
- **원인**: 일괄 적용 후 `renderResult(result)` 호출 → `renderReviewBar`가 `#rvFilter`를 innerHTML로 재생성 → 선택값 소실.
- **영향**: "미검토만 보기" 상태에서 일괄 처리하며 진행하는 흐름이 매번 끊김.
- **해결**: 재생성 전 `rvFilter` 값을 보존 후 복원하거나, 진행률 텍스트만 갱신하고 select는 재생성하지 않도록 분리.

### BUG-4. (발견 즉시 수정 완료) 대시보드 막대 채움 미표시
- **원인**: `.bar-fill`이 `<span>`(inline)이라 `width/height` 무시 → 트랙만 보이고 색 채움이 렌더되지 않았음.
- **조치**: `display:block` 추가로 수정 완료(이번 커밋). 스크린샷 재검증 완료.
- **교훈**: DOM 존재 여부만 확인하는 테스트는 시각 결함을 놓친다 → 스크린샷 검증을 테스트 절차에 포함할 것(TEST-1).

### BUG-5. 분석 취소가 실행 중인 단일 검사 내부에서는 반응하지 않음 【Major·구조 개선 필요】
- **원인**: 취소는 detector **사이**에서만 확인(`ANALYSIS_CANCELLED` 체크가 for 루프 선두). Worker는 동기 detector 실행 중 `cancel` 메시지를 이벤트 루프가 비어야 처리한다. 100k행 LOF(약 20초+) 중에는 취소 클릭 후 해당 검사가 끝나야 반영.
- **영향**: 대용량에서 "취소가 안 먹는다"고 느낌. 부분 결과 보존이라는 설계 의도와의 트레이드오프.
- **해결(2단계)**: ① LOF/DBSCAN의 청크 진행 콜백 지점(`(i & 4095)===0`)에서 워커 내 플래그 확인 후 예외로 탈출(해당 검사만 skipped 처리, 이전 결과 보존) — 즉시성 확보. ② UI에 "즉시 중단(결과 폐기)" = `worker.terminate()`+재생성 옵션 병행.

### BUG-6. 시각 포함 날짜의 시간대 불일치 가능성 — `Values.toDate` 【Minor·Edge】
- **원인**: CSV의 `"2024-06-01 13:00:00"`은 `new Date(문자열)`로 **로컬 시간** 파싱, xlsx 시리얼 날짜는 UTC 기준 변환. 날짜만 있는 값은 양쪽 모두 자정 UTC로 일치하지만, 시각이 포함되면 비교 검증에서 같은 시각이 "변경"으로 오판될 수 있다.
- **영향**: 시각 단위 시계열(센서 로그 등) 비교 시 오탐. 현재 샘플·주 사용례(일 단위)는 영향 없음.
- **해결**: `toDate` 문자열 파싱을 수동 분해 후 `Date.UTC()`로 통일.

### BUG-7. `reviewStorageKey`가 파일명+행×열 조합 【Minor】
- **원인**: 같은 이름·같은 크기의 다른 파일(예: 월별 동일 양식)에서 이전 검토 상태 복원을 물어봄. 요구사항의 "분석시각" 요소가 키에 없음.
- **영향**: 다른 데이터에 이전 검토가 복원될 수 있음(사용자가 확인창에서 거부 가능하므로 완화됨).
- **해결**: 키에 데이터 지문(예: 첫 행+마지막 행의 cellText 해시) 추가.

### 확인 결과 문제 없음(오탐 방지 기록)
- 이벤트 중복 등록: 대시보드/검토바/미리보기는 innerHTML 재생성 후 리스너 재부착 방식 — 요소가 교체되므로 중복·누수 없음. document 레벨 리스너(툴팁)는 1회 등록 확인.
- 메모리 누수: Worker 단일 인스턴스 재사용, blob URL 1회 생성, 다운로드 URL은 `revokeObjectURL` 처리 확인.
- Race: `btnRun` 비활성/`isUploading`/`busy` 플래그로 단일 비행 보장. 취소 후 재실행 시 `ANALYSIS_CANCELLED` 초기화가 `runAnalysis` 선두에 있어 안전.
- CSV 파서: 따옴표 내 개행/CR 보존, BOM 제거, EUC-KR 폴백 — 인용 필드 내 `\r`은 보존됨(코드 재확인).

## 3. 성능

### PERF-1. `renderHistogram`이 컬럼 변경마다 전 행 재스캔 【Minor】
- 100k행 × `toNumber` ≈ 수십 ms로 허용 범위. 단, 컬럼 선택을 연타하면 체감. `numericFrame`이 엔진에만 있고 UI에 없어 재계산함.
- **해결**: `lastResult`에 컬럼별 숫자 배열 캐시(첫 조회 시 생성) — 5줄 내 수정.

### PERF-2. 미리보기 테이블은 300행/40열 상한으로 방어됨 — 적정. 다만 `renderPreview`가 필터 변경마다 전체 행 순회(O(n))하는 것은 100k에서 수십 ms — 허용. 개선 여지: flags 인덱스(이상 행 목록) 사전 구축.

### PERF-3. ML 확장성(측정치 기록)
- 실측: 100k행 2열 — LOF 23s(정확), DBSCAN 9s(정확); 120k — 근사 모드 LOF 11s / DBSCAN 0.2s. brute 대비 LOF 동일 데이터 3k행 기준 수백 배 개선.
- 잔여 리스크: 고차원(7열 이상 숫자)은 자동 brute 폴백 → `mlRowLimit`(20만) 상한 내에서도 느릴 수 있음. **고차원 대용량 시 근사 모드로 우회하는 분기 추가 권장**(현재는 dim>6 & n>3000이면 brute).

### PERF-4. `duplicate()`가 전 행의 `cellText`를 매 분석마다 재생성 【Minor】
- O(n×m) 문자열 생성. 100k×20열이면 200만 문자열. 실측상 병목은 아니나, 형식 오류 검사·numericFrame과 함께 행 순회가 3회 중복된다. 장기적으로 단일 패스 통합 여지.

## 4. 코드 품질

### QUAL-1. `escapeHtml` 3중 정의 (2014 / 2839 / 3116행) 【Minor】
- XlsxWriter 내부 `esc`는 `'` 미이스케이프(속성에 안 쓰여 안전하나 불일치). 툴팁 `esc`, UI `escapeHtml` 각각 정의.
- **해결**: 엔진 스크립트(전역)에 `escapeHtml` 1개로 통합. 단, XlsxWriter의 esc는 XML 이스케이프 용도로 목적이 다르므로 이름을 `xmlEscape`로 바꿔 의도 구분.

### QUAL-2. 단일 파일 3,800줄 【장기】
- 내부망 배포(단일 HTML) 제약상 의도된 구조이며 섹션 주석으로 항해 가능. 그러나 diff 리뷰·충돌 위험이 커지고 있다.
- **해결(장기)**: 소스는 `src/*.js`로 분할하고 빌드 스크립트(Node 한 파일)로 index.html을 생성하는 개발 파이프라인. 배포 산출물은 지금과 동일한 단일 HTML 유지.

### QUAL-3. 매직 넘버 산재 【Minor】
- `LIMIT=300`, `MAX_COLS=40`, `BRUTE_MAX=3000`, `SPATIAL_MAX=100000`, `SAMPLE_SIZE=15000`, LOF 임계 `1.5` 등이 각 함수 내부에 있음. 엔진 상단 `const TUNING = {...}` 객체로 모아 운영 중 조정 용이하게.

### QUAL-4. `XlsReader` FORMULA 레코드의 `isText` 필드가 데드코드 (계산 후 미사용) — 삭제 권장. 【Minor】

### SOLID 관점
- Detectors 레지스트리(개방-폐쇄), 리더 3종의 공통 `open/grid/read` 인터페이스, XlsxWriter의 범용 `buildWorkbook` — 확장점이 잘 설계됨. 비식별 모듈(§별첨)도 동일 패턴으로 추가 가능함을 확인.

## 5. 아키텍처

- **응집도/결합도**: 엔진은 DOM 무의존(워커 재사용의 근거), UI는 전역 노출(`window.__anomaly/__upload/__readers/__compare`)로 테스트 접점 제공 — 양호.
- **의존 방향**: UI → 엔진/리더/작성기 단방향. 비교 모듈이 `window.__upload.validateUploadFile`을 통해 검증 공유 — 순환 없음.
- **향후 기능 추가 시 문제점**: ① 모드(탭)가 3개 이상 되면 상단 탭+긴 세로 폼 구조의 항해성이 급락(→ 레이아웃 리뷰 §7). ② 결과 상태(lastResult/reviewState)가 IIFE 클로저에 갇혀 모드 간 공유 불가 — 비식별 모듈이 "분석 결과를 이어받아 처리"하려면 명시적 상태 전달 통로 필요.

## 7. UX / UI (엔터프라이즈 관점)

### 잘된 점(짧게)
파스텔+진한 글자(전 조합 WCAG AA 13~15:1 실측), 스텝 인디케이터, 셀 단위 하이라이트, 근거 툴팁 — 업무 시스템 문법에 부합.

### 문제점
- **UX-1 【Major】 정보 밀도 대비 세로 스크롤 과다**: 단일 분석 화면이 파일선택→옵션→허용컬럼→파라미터→숫자컬럼→저장→결과→실행바 순으로 1열 나열. FHD에서 분석 시작 버튼까지 3~4 스크롤. ERP형 도구는 "설정(좌) | 결과(우)" 2열 또는 섹션 접힘이 표준. → 개선안: 설정 패널들을 접이식(현재 대시보드처럼)으로 만들고, 파일 로드 후 자동으로 '분석 옵션'만 펼침.
- **UX-2 【Minor】 실행 버튼과 결과의 거리**: 실행바(sticky bottom)는 좋으나, 분석 완료 후 결과 패널로 자동 스크롤이 없어 완료를 인지하기 어려움 → `resultPanel.scrollIntoView()` 1줄.
- **UX-3 【Minor】 오류 메시지 톤 일관**: 업로드 계열은 친절한 존댓말인데 `alert("분석 중 문제가 발생했습니다…")`는 브라우저 기본 alert — 시스템 느낌을 깨뜨림. 인라인 배너(`skipNote` 스타일 재사용)로 통일 권장.
- **UX-4 【Minor】 접근성**: 결과 테이블 내 `select.rv-status`/`input.rv-memo`/`input.rv-check`에 aria-label 없음(스크린리더가 "콤보박스"로만 읽음). 스텝 인디케이터에 `aria-current="step"` 부재.
- **UX-5 【Minor】 버전/빌드 표기 부재**: 화면 어디에도 버전이 없어 운영 중 "어느 배포본인지" 확인 불가(§9 OPS-1과 동일 항목).
- **디자인 원칙 점검**: Gradient는 헤더 1곳(기존 사용자 수정 존중), glassmorphism/과한 shadow/floating card 없음, radius 6~10px 수준 — 지양 목록 위반 없음. 현행 유지.

## 8. 레이아웃 검토 — 사이드바 판단

**결론: 현재(2개 모드)는 Case 2 — 사이드바 미도입. 단, 비식별 처리 모듈이 추가되어 3개 모드가 되는 시점에 Case 1로 전환(슬림 좌측 레일)을 권고.**

근거:
1. 참고 사이트(data.vaiv.kr)는 데이터 **카탈로그/포털**(다수 페이지·검색·분류 탐색)이라 사이드바가 필수 구조다. 본 도구는 "업로드→설정→실행→결과"의 **단일 작업 흐름 도구**로, 현재 2개 모드는 상단 탭이 인지 비용이 가장 낮다.
2. 사이드바는 가로 200px+를 상시 소모한다. 본 도구의 병목은 **결과 테이블 가로 폭**(수십 컬럼 미리보기)이므로, 모드가 2개인 지금 사이드바는 손해가 더 크다.
3. 그러나 모드가 3개(단일 분석/비교 검증/비식별 처리)가 되면 상단 탭 폭이 좁아지고, 향후 "설정/이력" 등 부가 메뉴 확장도 예상된다. 이 시점에는 **아이콘+라벨 슬림 레일(펼침 180px/접힘 52px, 접힘 상태 기본)**이 적합하다 — 구현 프롬프트(§opus-implementation-prompt.md)에 전환 사양 포함.

## 별첨 — 개인정보 비식별 처리 기능 설계 (요약)

> 상세 구현 지시는 `opus-implementation-prompt.md` §C 참조. 여기서는 설계 판단만 기록.

### A. 배치와 구조
- **독립 모듈**: 신규 모드 "비식별 처리"(3번째 탭 → 사이드바 전환 트리거). 기존 리더(`loadWorkbook`)·작성기(`XlsxWriter.buildWorkbook`)·검증(`validateUploadFile`)을 재사용하고, 탐지/변환 엔진(`DeidEngine`)은 engineScript에 DOM 무의존으로 추가(대용량 시 Worker 재사용 가능).
- **원본 불변**: 변환은 새 2차원 배열 생성. 다운로드 시트는 `비식별결과` + `처리요약`(처리 컬럼/방식/건수/제외 컬럼) 2장.

### B. 자동 탐지 — 이중 신호 + 신뢰도
1) **컬럼명 사전**(한국 공공/산단 실무 관행): 성명/이름/대표자/담당자, 주민등록번호, 외국인등록번호, 운전면허, 여권, 휴대전화/핸드폰/연락처, 전화, 이메일/메일, 주소/소재지, 생년월일/생일, 사업자등록번호/사업자번호, 법인등록번호, 계좌, 카드, IP.
2) **값 패턴 정규식**(비결측 표본 최대 1,000행 매칭률): 주민번호 `\d{6}-?[1-4]\d{6}`, 휴대폰 `01[016789]-?\d{3,4}-?\d{4}`, 이메일, 사업자 `\d{3}-?\d{2}-?\d{5}`, 법인 `\d{6}-?\d{7}`, 카드 `\d{4}(-?\d{4}){3}`(+Luhn), 계좌(숫자 10~14 연속, 은행명 병기 시 가중), IP, 생년월일(YYYYMMDD/YYYY-MM-DD 범위 검증), 한국 성명(`^[가-힣]{2,4}$` — 지역명 등 오탐 여지 있어 컬럼명 신호와 결합할 때만 채택).
3) 신뢰도 = 컬럼명 신호(0.5) + 패턴 매칭률(0.5×비율) → 상/중/하 표시, 기본 체크는 '상'만.

### C. 방식별 적합성 판단 (도메인 반영)
| 방식 | 적용 | 판단 |
|---|---|---|
| 마스킹 | 성명(홍*동), 전화(010-****-1234), 사업자번호(123-**-*****), 이메일(ab***@도메인) | **기본값**. 형식 보존으로 업무 판독성 유지 |
| 삭제 | 주민번호 등 고위험 식별자 | 권장 기본값(주민/외국인/여권/면허) |
| 가명처리 | 담당자명 등 조인 키로 쓰이는 값 | 동일 원본→동일 가명(담당자001) 매핑 유지로 파일 간 연결성 보존 |
| 일반화 | 생년월일→연령대, 주소→시·군·구 절단 | 통계 활용 유지에 적합 |
| 해시 | 내부 대조용 키가 필요한 경우만 | **솔트 필수 입력** + "복원 불가" 경고. 전화번호처럼 전수 탐색 가능한 값은 해시 단독으로는 재식별 위험이 있음을 UI에 명시. Web Crypto 의존 없이 자체 SHA-256 구현(오프라인·file:// 확실성) |
| (부적합) 잡음 추가·총계처리 | 수치 개인정보가 주가 아닌 본 도구 특성상 제외 — 필요 시 일반화로 대체 |

- 도메인 주의: **기업명·공장명·산단명은 원칙적으로 개인정보가 아님** → 자동 탐지에서 제외하되, 개인사업자 상호(성명 포함 가능)를 위해 수동 선택은 허용. **사업자등록번호는 개인사업자의 경우 개인정보에 준함** → '중' 신뢰도로 기본 제안.

### D. 성능·충돌 방지
- 변환은 컬럼 단위 O(n) 문자열 치환 — 10만 행 무리 없음(가명 매핑은 Map). 기존 분석/비교 상태와 완전 분리(자체 workbook 상태), 기존 전역·CSS 네임스페이스와 충돌하는 식별자 없음(`deid-` 접두).
